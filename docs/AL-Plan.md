# Algorithm Alignment Evaluation

The current application (comprising `autonomous_agent.py` and `GUI-auto_improve_experiments.py`) attempts to capture the spirit of automated improvement, but structurally, it functions closer to a simple hyperparameter tuning system rather than the sophisticated agentic workflow detailed in `ALGORITH.md`.

While it operates a loop to execute python scripts and collects metrics, it lacks the core pillars defined in the reference:

1. **No True Human Layer:** There is a GUI, but it serves mainly as a script runner and metric visualizer. The critical RL "Review & Decision" step (Approve/Modify/Reject of AI plans) is bypassed.

2. **Missing Skill Chaining & Code Modification:** The current agent simply mutates predefined numeric parameters using `OptimizationStrategy` and regex math (`suggest_modifications`). It does not autonomously read logs, identify bugs, generate abstract code modifications, or chain skills (e.g., `debug -> fix -> report`).

3. **Absence of Cognitive Memory:** While it tracks high scores in `optimization_results.json`, it does not extract or embed "successful strategies" or "failure modes" into an indexed knowledge base as required by the algorithm.

4. **Rudimentary Planning:** The Plan step is relegated to tweaking parameters randomly within brackets instead of generating formalized hypotheses, success metrics, constraints, and execution sequences for human review.

To fully realize the goals set out in `ALGORITH.md`, the platform requires a structural rewrite to transition from a hyperparameter tuner to an LLM-orchestrated coding agent framework.

## Algorithm Alignment Implementation Plan

This plan details actionable, step-by-step instructions to bridge the gap between the current state and the closed-loop system defined in `ALGORITH.md`.

## Phase 1: Establish the Data Models and Memory Foundation

To move away from basic numeric tuning, the system must be capable of tracking high-level abstractions like "Hypothesis" and "Knowledge Pattern".

1. **Implement Core Data Classes:**
   - Define `ExperimentPlan(hypothesis, metrics, constraints, steps)` logic inside `autonomous_agent.py`.
   - Define `ExecutionReport(summary, metric_deltas, root_causes, suggested_fixes)`.

2. **Setup Extensible Persistent Memory (`update_memory`):**
   - Create a vector database or an indexed JSON/SQLite schema (`memory_store.py`) to hold textual strategies, past successes, and documented failures.
   - Integrate an embedding function or indexing system to allow the agent to recall past failure modes (e.g., "Increasing base learning rate beyond 0.3 always caused memory corruption").

## Phase 2: Build the Agent Execution Engine & Skill Chains

The agent must be capable of executing multi-step cognitive workflows, not just randomizing configs.

1. **Implement the Skill Chaining Engine:**
   - Develop atomic functions/tools for the agent: `run_skill_chain(input_data, skills_list)`.
   - Setup an LLM inference integration (if not present) that acts as the "Agent Engine".
   - Define the standard tasks: `job-debug()` (reads tracebacks), `issue-fix()` (outputs git patches), `issue-report()` (explains metrics).

2. **Enhance Execution Pipeline:**
   - Update `execute_experiment()` to use the Agent Engine to optionally refactor code logic, validate compilation/flake8, and invoke `standard_apgi_runner.py`.

## Phase 3: Implement the RL Experiment Loop

Wire up the complete end-to-end continuous loop.

1. **Introduce the Plan Stage (`plan_experiment`):**
   - The Agent consumes a `task` requirement and writes an `ExperimentPlan`.

2. **Update the Analyze Stage (`analyze_results`):**
   - Modify the subprocess response parser to allow the LLM to write a natural language summary of the results rather than merely extracting regex metrics. Provide it the delta against the baseline in memory.

3. **Build the Iteration Logic (`iterate`):**
   - If an experiment fails the metrics, route the `ExecutionReport` back into the Agent Engine to plan a fix (`modify`/`reject`).
   - If it succeeds, update memory constraints and log the victory.

## Phase 4: Construct the Human Review GUI (Control Layer)

Rewrite the GUI to function as the steering mechanism and checkpoint guardrail.

1. **Build the Hypothesis Approval Board:**
   - Instead of a generic "Run" button, the GUI must display pending `ExperimentPlans` generated by the agent.
   - Provide human intervention logic: `human.review(report)` outputting `{approve, modify, reject}`.

2. **Implement Guardrail System Controls:**
   - Create a dashboard for Escalation Triggers (e.g., "Stop AI if confidence < 70% or regression > 5%").
   - When guardrails are triggered, pause the `Auto-Loop` and queue a notification for the researcher in the GUI.

## Phase 5: Integrate Output & Autonomy Logic

1. **Complete Auto-Loop Logic:****
   - Refine `Auto-Loop Condition` logic. If no constraints are violated and metric confidence is high, allow the LLM execution pipeline to skip the GUI pause and commit code to git automatically via `async_commit`.
2. **Final Validation:**
   - Create a dedicated `test_rl_loop.py` mapping out each stage. Ensure an intentional failure can trigger the debug-fix skill chain and learn from its mistake.
